# AGENT.md — Snapshot 2026-05-19

## Changes in this version

- **GFE continuation paywall fully wired** — `gfe_message_count` now increments on every fan message in `orchestrator.py`. Previously the counter existed but was never incremented, so the paywall never fired.
- **Continuation threshold raised: 25–35 → 40–50 messages** — less aggressive, gives fans more breathing room before hitting the paywall.
- **Continuation timing is now agent-decided, not mechanical** — orchestrator sets `gfe_continuation_ready: True` in context when threshold is hit; agent decides whether the moment is socially right to send it. If it skips, the flag stays set next turn. Agent signals via `ppv: {"tier": "continuation"}` in its JSON output.
- **Fixed crash** — `"continuation"` tier value is now handled before the `int(ppv_tier)` cast in the orchestrator PPV block, preventing a `ValueError` if the agent returned it.

# Massi-Bot — Agent Technical Reference

Read this at the start of every session. It tells you what the system does, where everything lives, and why decisions were made the way they were. Update it whenever something significant changes.

---

## What this system does

An automated chatbot that manages conversations with fans on Fanvue (and optionally OnlyFans). It builds rapport, detects sexual intent, drops PPV content tiers progressively ($27–$200), handles custom content orders, and maintains a GFE (girlfriend experience) relationship with each fan. Revenue model: 6-tier PPV ladder, custom orders, and a $20 GFE continuation paywall every ~40–50 messages.

---

## Architecture — single agent

**One Opus 4.7 call per fan message** via OpenRouter. No multi-agent pipeline, no scripted templates, no state machine routing.

```
Fan message
    → fanvue_connector.py (webhook / inbox poll)
        → orchestrator.py (build context, call agent, post-process)
            → single_agent.py (Opus 4.7 with tool use)
                → uncensor_agent.py (Grok — auto-called when horniness > 5)
        → parallel_guardrails.py (8 concurrent checks, zero latency)
        → execute_actions() (send messages, drop PPVs, with delays)
```

**Tools the agent can call** (Opus decides, not forced):
- `uncensor(text, tier)` — replaced with auto post-processing; tool still exists but rarely used
- `classify_custom_request(request_text, custom_type)` — prices a custom order, creates `pending_custom_order`
- `fire_custom_payment_alert(reason)` — Telegram alert to admin for custom payment verification
- `get_specific_memories(query)` — RAG memory lookup

**Auto post-processing** (runs AFTER Opus, not tools):
- Grok uncensor kicks in automatically when `horniness_score > 5` — Opus never self-censors anymore
- 8 parallel guardrails (tier boundary, no-redrop, emoji rate, passive voice, etc.)
- PPV heads-up injection + Cobalt-Strike jitter (108–252s between heads-up and PPV drop)
- Message deduplication (drops consecutive identical outputs from Opus)
- `_split_into_chunks()` — splits reply into natural thought segments sent with 1.2–2.5s delays

## Subscriber state — what's stored where

All subscriber state serializes into **two Supabase columns**:

**Dedicated columns** (fast filtering): `username`, `display_name`, `state`, `whale_score`, `total_spent`, `persona_id`, `loop_count`, `callback_references`, `recent_messages`, `last_message_at`

**`qualifying_data` JSONB** (everything else): horniness_score, fan_name, fan_profile, tags, pending_ppv, pending_custom_order, sext_consent_given, gfe_message_count, high_value_utterances, last_successful_bot_message_at, recovery state, session number, all timestamps

**Rule: never add a new DB column. Always use `qualifying_data`.** The store handles it — just add the field to `Subscriber`, serialize in `_subscriber_to_row`, deserialize in `_row_to_subscriber`.

## Key subsystems

### Horniness score (0–10)
Replaces the old binary `sext_consent_given` gate. Updated every message.

- Opus scores it in JSON output (`horniness_score` field)
- Code-level keyword detector overrides conservative Opus scoring: explicit words → 9, sexual interest words → 6, cooldown words → −3
- Final = `max(opus_score, keyword_boost)` except cooldown which applies as delta
- **Grok auto-triggers when score > 5** — no tool call needed, post-processing in `single_agent.py`
- `sext_consent_given=True` (legacy) → score loads as `max(8, stored_score)` for backwards compat
- **Time decay on load** (`subscriber_store.py:_decay_horniness`): gap < 1h = no decay, 1–4h = −2, 4–24h = ×0.5, 24–72h = ×0.3, 72h+ = ×0.2. Floor: 3 if ppv_count > 0, else 0

### Fan profile
Persistent cross-session knowledge about each fan: `personality`, `interests` (list), `kinks` (list), `notes`.

- Populated two ways:
  1. **Opus real-time**: agent sets `fan_profile_update` in JSON output when it learns something new this turn (partial update, lists are new-items-only)
  2. **Haiku background**: every 10 fan messages, `llm/fan_profiler.py` runs a dedicated extraction pass (temp=0.1, conservative, no guessing), merges result into subscriber
- Orchestrator merges: strings overwrite, lists dedupe-append (case-insensitive)
- System prompt shows profile with **fan-specific usage rules** when non-empty: kinks → act on them during escalation, interests → reference during rapport/cooldown, personality → calibrate pacing, notes → situational context

### Fan name
- Agent sets `fan_name` in JSON output only when fan first tells it this turn
- Stored on `sub.fan_name`, persisted in `qualifying_data`
- System prompt: if unknown → ask naturally in first 2–3 messages, once only; if known → use during escalation, not every message

### Recovery mode (server outage)
- Inbox poll detects messages older than 15 min → `_recovery_contexts[fan_uuid] = {bot_gap_str, msg_count}`
- One Telegram alert fires per startup cycle: how many fans, max gap
- Recovery context passed to first orchestrator call only (not regen/post-send passes)
- System prompt tells agent: respond in your voice, no forced apology
- **PPV suppressed on first reply back** — next fan message resumes normal selling
- Grok/uncensor and horniness score run normally — tone stays matched to relationship history
- Stale fans get 3s stagger between webhook injections to avoid rate limiting

### Message splitting
`_split_into_chunks()` in `fanvue_connector.py` — pure regex, no LLM call.
- Splits on: `...`, sentence-ending punctuation before whitespace, `\n`
- Merges only genuine orphans (< 3 chars)
- Chunks sent with 1.2–2.5s random delays between them
- 62 tests in `tests/test_split_chunks.py`

### Inbox poll
- Runs every 30s as fallback for missed webhooks
- Deduplicates by `model_id` (not `creator_uuid`) to avoid 10x firing from UUID alias accumulation
- Poll size: 100 chats
- Injects unread chats as signed internal webhooks to `localhost:8000/webhook/fanvue/`

### PPV flow
1. Agent decides to drop PPV → includes `ppv` field in JSON output
2. Orchestrator reads tier, looks up price from model profile (or defaults)
3. If heads_up text provided: sends it, then waits 108–252s (Cobalt jitter) before PPV
4. PPV sent via `send_fanvue_ppv()` — price in CENTS (×100)
5. `pending_ppv` tracked on subscriber for 6h auto-delete if unpaid
6. **OnlyFans prices are in DOLLARS — do not multiply**

### Custom orders
1. Fan requests something specific → agent calls `classify_custom_request` tool
2. Tool prices it from WILLS_AND_WONTS pricing, creates `sub.pending_custom_order`
3. Agent quotes price, fan confirms → agent calls `fire_custom_payment_alert`
4. Admin gets Telegram inline keyboard (Confirm / Deny)
5. On confirm → bot tells fan "deliver within 48 hours"
6. Custom PPV is a continuation-photo placeholder sent at the quoted price

### GFE continuation paywall
- Every ~40–50 messages (jitter: 40–50), if no tier PPVs purchased, bot sends a $20 continuation paywall
- Content: clothed lifestyle photos from `continuation/` folder (tier=0 in content catalog)
- **Counter:** `gfe_message_count` increments on every fan message in `orchestrator.py`
- **Timing is agent-decided, not mechanical.** When count ≥ threshold, orchestrator sets `gfe_continuation_ready: True` in context. The agent sees this and decides whether the current moment is right (good: natural pause, warm tone / bad: mid-escalation, fan venting, about to leave). If it skips, the flag stays set next turn.
- Agent signals intent via `ppv: {"tier": "continuation"}` in its JSON output
- After payment: `gfe_message_count` resets, threshold re-jitters to 40–50
- Suppressed entirely once the fan has purchased any tier PPV (he's in the selling funnel)

## Pricing rules — CRITICAL

- **Fanvue: prices in CENTS.** `$27.38 → 2738`. Always `round(price * 100)` before sending.
- **OnlyFans: prices in DOLLARS.** Pass through unchanged.
- Default tier prices (in orchestrator, overridden by model profile):
  - Tier 1: $27.38 | Tier 2: $36.56 | Tier 3: $77.35
  - Tier 4: $92.46 | Tier 5: $127.45 | Tier 6: $200.00
- Odd cents are intentional — feels human-set, not corporate

## Things that look wrong but aren't

- **`sext_consent_given` still exists** — legacy backwards-compat. New logic uses `horniness_score > 5`. Both coexist.
- **Grok is never called as a tool** — it was, but Opus always self-reported as explicit enough. Now it auto-runs in post-processing when horniness > 5. The `uncensor` tool definition still exists for Opus to optionally call, but the real trigger is code-level.
- **`_model_contexts` has multiple keys for the same model** — Fanvue uses different UUIDs in different contexts (recipientUuid aliases). The inbox poll deduplicates by `model_id` before iterating. Don't "fix" this by collapsing the dict.
- **`qualifying_data` JSONB holds everything** — this is intentional. No schema migrations for new subscriber fields, just add to the store. The dict grows over time; that's fine.
- **Message settle window is 8s+** — this is intentional. Fans often send 2–3 rapid messages. The settle window batches them into one Opus call instead of firing 3 separate replies.
- **PPV jitter is 108–252 seconds** — intentional "Cobalt-Strike jitter" to make the delay between heads-up and PPV drop feel human and unpredictable.

## Do not touch without understanding

- `verify_signature()` in `fanvue_connector.py` — HMAC verification. Never bypass or weaken.
- `_processed_message_uuids` dedup cache — prevents double-processing from webhook + poll overlap. Capped at 500 entries.
- `_get_sub_lock()` per-fan asyncio lock — prevents concurrent handlers racing on the same fan. Every message path must go through this.
- `save_subscriber()` call order — always save BEFORE `execute_actions()`. PPV send can take 30–240s; subscriber state must persist before any delay.
- Price × 100 for Fanvue — enforced at `execute_actions()` send time. Do not change without testing.
